## 一、结构化日志系统

**文件**: `trainer/logger.py`

### 为什么替换原始的 `print()`

原始 Logger 直接 `print(content)` → 无时间戳、无日志级别、无来源模块、无颜色区分、无文件持久化。生产环境中排查问题时，你无法知道哪个错误先发生、在哪个模块、什么级别。

### 新架构

```
get_logger(name) → Python logging.Logger (minimind.<name>)
    ├── StreamHandler → RichHandler (彩色终端, traceback 美化的)
    ├── FileHandler   → 可选 (MINIMIND_LOG_FILE 环境变量)
    └── Logger() shim → 委托到 .info() (旧代码兼容)
```

### 基础用法

```python
from trainer.logger import get_logger

log = get_logger(__name__)  # 建议传入 __name__

log.debug("详细的调试信息, 仅 DEBUG 级别输出")
log.info("训练开始, batch_size=%d, epochs=%d", batch_size, epochs)
log.warning("GPU 显存使用率超过 80%%")
log.error("Checkpoint 保存失败", exc_info=True)  # 打印 traceback
```

### 运行时控制

```python
from trainer.logger import set_level

set_level("DEBUG")   # 切换为调试级别
set_level("WARNING") # 仅显示警告及以上
```

### 环境变量

```bash
# 启用文件日志 (所有输出同时写入文件)
MINIMIND_LOG_FILE=./training.log MINIMIND_LOG_LEVEL=DEBUG python train_full_sft.py
```

| 变量 | 可选值 | 默认 | 说明 |
|------|--------|------|------|
| `MINIMIND_LOG_LEVEL` | `DEBUG`, `INFO`, `WARNING`, `ERROR` | `INFO` | 日志级别 |
| `MINIMIND_LOG_FILE` | 任意文件路径 | 空 (不写文件) | 文件输出路径 |

### 向后兼容

旧代码中遍布的 `Logger(f"...")` 调用保持原样即可，内部委托到 `logger.info()`：

```python
# 旧代码 — 无需修改
from trainer.trainer_utils import Logger
Logger("训练开始")  # → [INFO] 训练开始

# 等价于新写法
from trainer.logger import get_logger
log = get_logger("minimind")
log.info("训练开始")
```

### RichHandler 效果

有 Rich 库时 (已安装)，终端输出效果：
- **颜色**: DEBUG=灰色, INFO=白色, WARNING=黄色, ERROR=红色
- **Traceback**: 语法高亮 + 行号
- **时间戳**: 精确到秒
- **自动换行**: 长日志自动折行

没有 Rich 时自动降级为纯文本 `%(asctime)s | %(levelname)-7s | %(name)s | %(message)s` 格式。

## 二、YAML 配置系统

**文件**: `trainer/config.py` + **根目录 `config.yaml`**

### 为什么需要配置系统

原始 MiniMind 所有超参数散落在 8 个训练脚本的 `argparse` 定义中：
- 改动参数需要改代码
- 不同脚本间的默认值不一致
- 无法保存/复用配置
- 无法做配置版本管理

新系统将 **所有 60+ 超参数集中到 `config.yaml`**，支持：直接修改 → 训练即生效；YAML 版本控制；多实验配置并存。

### 三层合并策略

```
Layer 1: config.yaml 内置默认 (所有参数都有合理默认值)
    ↓ 覆盖
Layer 2: 用户自定义 YAML (--config my_experiment.yaml)
    ↓ 覆盖
Layer 3: CLI 参数 (--epochs 5 --batch_size 64)
    ↓
最终 TrainingConfig 实例
```

### 使用方式

```bash
# 方式1: 只修改 config.yaml → 直接运行
vim config.yaml
python trainer/train_full_sft.py

# 方式2: CLI 临时覆盖 (不改文件)
python trainer/train_full_sft.py --epochs 10 --batch_size 128

# 方式3: 多实验管理
python trainer/train_full_sft.py --config experiments/exp1_small.yaml
python trainer/train_full_sft.py --config experiments/exp2_large.yaml
```

### Python API

```python
from trainer.config import load_config, TrainingConfig, create_arg_parser

# 三层合并
config = load_config("full_sft", sys.argv[1:], user_yaml="my_experiment.yaml")

# TrainingConfig 是标准 dataclass — IDE 有补全
print(config.learning_rate)    # 1e-05
print(config.batch_size)       # 32
print(config.hidden_size)      # 768

# config.auto_device() — 惰性设备检测 (不强制 import torch)
device = config.auto_device()  # "cuda:0" / "cpu"

# create_arg_parser() — 向后兼容的 argparse
parser = create_arg_parser("full_sft")
args = parser.parse_args()
```

## 四、Checkpoint 管理器

**文件**: `trainer/checkpoint.py`

### 为什么需要新的 Checkpoint 管理器

原始的 `lm_checkpoint()` 使用 `torch.save` (pickle 格式)，存在安全隐患和互操作性问题。新的 `LMCheckpointer` 默认使用 `safetensors` (安全的零拷贝格式)，同时提供自动轮转和元数据记录。

### 基础用法

```python
from trainer.checkpoint import LMCheckpointer

ckp = LMCheckpointer(
    save_dir="../checkpoints",
    max_keep=3,           # 最多保留 3 个 checkpoint
    fmt="safetensors"     # "safetensors" (推荐) / "torch"
)

# 保存 — 原子写入
ckp.save(
    model=model,
    optimizer=optimizer,
    epoch=0,
    step=100,
    weight_name="full_sft",
    lora_state=None,        # 可选: LoRA 权重
    extra={"experiment": 3, "notes": "baseline run"}
)
# 写入: full_sft.safetensors + full_sft_meta.json + full_sft_optim.pth

# 加载 — 自动尝试 safetensors → torch
state_dict = ckp.load("full_sft")

# 只读元数据 (不加载权重)
meta = ckp.load_metadata("full_sft")
# → {"epoch": 0, "step": 100, "config_hash": "a3f8c2e1", "timestamp": 1780152000}

# 列出所有 checkpoint
for name, step, ts in ckp.list_checkpoints():
    print(f"{name}: step={step}, ts={ts}")
```

### 写入过程详解

```
save() 被调用
  ├── 1. 解包模型 (DDP/compile wrapper)
  ├── 2. state_dict 转 CPU
  ├── 3. 写入 .tmp 临时文件 (safetensors 或 torch)
  ├── 4. os.replace(.tmp → 正式文件)  ← 原子操作, 崩溃安全
  ├── 5. 写入 _meta.json (epoch, step, config_hash, timestamp)
  ├── 6. 写入 _optim.pth (优化器状态, 始终 torch 格式)
  ├── 7. (可选) 写入 _lora.pth
  └── 8. _rotate() — 如果超过 max_keep, 删除最旧的
```

### Checkpoint 轮转算法

```
_rotate("full_sft"):
  1. glob *_meta.json → 按 mtime 排序
  2. 计算总数 = 去重后的 base_name 数
  3. 目标删除数 = max(0, 总数 - max_keep)
  4. 跳过当前的 full_sft
  5. 从最旧的开始删 (meta + .safetensors/.pth + _optim.pth + _lora.pth)
```

### 对比旧版

| 特性 | 旧 `lm_checkpoint()` | 新 `LMCheckpointer` |
|------|---------------------|---------------------|
| 保存格式 | `.pth` (torch pickle) | `.safetensors` (默认) / `.pth` |
| 安全性 | pickle 可执行任意代码 | safetensors 纯数据 |
| 元数据 | 无 | `_meta.json` (epoch/step/hash/time) |
| 多格式 | 只读 torch | 自动尝试 safetensors → torch |
| 轮转清理 | 手动删除 | 自动 (max_keep) |
| 旧代码兼容 | — | `lm_checkpoint()` 仍可用 |

## 五、异常体系

**文件**: `trainer/exceptions.py`

### 层级结构

```
MiniMindError (基类)
  ├── ConfigError         — 配置错误 (未知任务类型、无效参数值)
  ├── CheckpointError     — 检查点错误 (文件损坏、找不到文件)
  ├── DataLoadError       — 数据加载错误 (文件缺失、格式无效)
  ├── ModelBuildError     — 模型构建错误 (架构参数无效)
  └── TrainingError       — 运行时错误 (NaN loss、OOM)
```

### 使用

```python
from trainer.exceptions import (
    MiniMindError, ConfigError, CheckpointError,
    DataLoadError, ModelBuildError, TrainingError,
)

# 配置验证
if task not in ["pretrain", "full_sft", "lora", "dpo", "grpo", "ppo", "agent", "distillation"]:
    raise ConfigError(f"Unknown task type: '{task}'")

# Checkpoint 错误
if not path.exists():
    raise CheckpointError(f"Checkpoint not found: {path}")

# 数据错误
if len(dataset) == 0:
    raise DataLoadError(f"Dataset is empty: {data_path}")
```

## 六、训练指标 CSV 导出

**文件**: `trainer/metrics.py`

### 用途

训练脚本在 `log_interval` 处写入一行 CSV，Jupyter notebook 实时读取并绘图。

### API

```python
from trainer.metrics import MetricsWriter

writer = MetricsWriter("./metrics.csv", flush_every=1)

# 每 log_interval 步调用一次
writer.write(step=100, loss=2.34, logits_loss=2.30, aux_loss=0.04, lr=0.00000999)
```

输出 (`metrics.csv`):

```csv
timestamp,step,loss,logits_loss,aux_loss,lr
1780152418.099,100,2.3400,2.3000,0.0400,0.00001000
1780152420.512,200,2.1500,2.1200,0.0300,0.00000999
```

`train_full_sft.py` 已集成。其他训练脚本可参照同样模式集成。
